เห็น before + delta + after · ก่อนกด

  • Delta box เป็นหัวใจของ fix — UI query v_stock_unified current qty · บวก qty ที่น้องพิมพ์ · โชว์ "5 + 3 = 8 kg" ชัดๆ · น้องเห็นว่ากำลังจะเกิดอะไรก่อน commit
  • Anomaly flag — ถ้า delta > 3× current qty หรือ > 50 kg สำหรับ item ที่ไม่เคยรับเยอะ → แสดง warning ขอให้ confirm ด้วย text input "พิมพ์ 'ยืนยัน' เพื่อส่ง" · ดัก user error ก่อน trigger ยิง sm
  • ผักบุ้ง SP-128 ของค้าง 12 kg — ตัวอย่างจริงจาก blueprint (22/04 workaround · zero-out) · ถ้าของจริงมีค้าง delta preview เตือน + เสนอ action "zero-out แล้วรับใหม่" เป็น option · ไม่ต้องวิ่งหา admin ยิง SQL
  • Keypad ในหน้าเดียว — ไม่ต้องเปิด system keyboard · ทั้งหน้าใช้มือถือเปียก · ปุ่ม 56px+ มือเอื้อมถึง
  • RPC contract ไม่เปลี่ยน — ส่ง p_qty = 3 (amount received) ให้ RPC · ปัญหาจริงอยู่ที่ trigger · UI ไม่แก้ root cause แต่ ลดผลกระทบเหลือ 0 กรณี เพราะน้องเห็นความไม่ถูกต้อง before commit

B1 · ทำไมต้องมี UI Guard

ปัญหาจริงอยู่ใน trigger · ไม่ใช่ UI · แต่ขณะที่ rpc_receive_universal ยังส่ง NEW.qty เป็น running total เข้า trigger, UI Guard ช่วยให้น้องไม่ commit transaction ที่จะพอง — เหมือน defensive driver ไม่ใช่ defensive road

ก่อน po-receive.html ปัจจุบัน

// น้องกดรับผักบุ้ง 3 kg · ของค้าง 12 client.rpc("receive_universal", { p_item: "SP-128", p_qty: 3 // amount received }) # ใน trigger: emit_sm_from_stock_counts: event = "receive" qty_delta = NEW.qty // = 3 (ถูก) # v_stock_unified หลัง insert: ✓ SP-128 = 15 kg (12 + 3) # แต่ถ้า client เผลอส่ง p_qty = running total 15: emit_sm_from_stock_counts: qty_delta = 15 // inflates! ✗ SP-128 = 27 kg (12 + 15)
