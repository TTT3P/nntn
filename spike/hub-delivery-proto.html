มองย้อน — 1 การ์ด = 1 trip ไม่ใช่ 4 rows

  • Group by date + branch + actor — query view ใหม่ v_delivery_trips รวม bill_no ใน window เดียวกัน (±30 นาที, สาขาเดียว, คนส่งคนเดียว) → UI เห็น 1 trip
  • แสดง bill_no จริงใน sub-list — ไม่ซ่อนหลัง UI เพราะ ops ต้อง reconcile กับใบส่งของลูกค้าตัวจริง (FoodStory portal เห็น 4 PO) · สามารถ tap ทุก row เปิด detail ของเดิม
  • "DB raw" expander · ไว้ให้ ไทน์ / dev debug · โชว์ 4 rows จริงจาก stock_counts + deliveries · เป็น transparency tool ไม่ใช่ user flow หลัก
  • หน้า 47→25 count: ตัด page แยก "ประวัติเนื้อ" กับ "ประวัติ nm" รวมเป็น 1 — เพราะ trip เดียวไม่ควรดู 2 ที่
  • Impact: ลูกค้าเจอบิลกระจาย 4 ใบเหมือนเดิม (constraint DB ยังอยู่) แต่น้อง NNTN operate ผ่าน 1 entity = feel "one trip" = ไม่ลืมเช็ค bill_no ใดไปในอีก tab

B2 fix — 1 บรรทัดของ client code

ใน submitDelivery() หลัง await rpcSubmitDelivery() ถ้า success → await supabase.from('delivery_drafts').delete().eq('bill_no', draftBillNo) · ถ้า re-open หน้า hub-delivery แล้ว query delivery_drafts WHERE bill_no NOT IN (SELECT bill_no FROM deliveries) → ได้ orphan drafts โชว์ banner "draft ค้างจากเมื่อวาน" · ให้ตัวเลือกทำต่อหรือล้าง

B6 fix — 1 view ใหม่

v_delivery_trips = SELECT branch, date(created_at), actor, SUM(qty), ARRAY_AGG(bill_no) FROM deliveries GROUP BY 1,2,3 · UI consume view นี้สำหรับ history list · รายละเอียด bill_no เดิมยังใช้ได้จาก deliveries ตรงๆ · ไม่แตะ bill_no UNIQUE constraint
