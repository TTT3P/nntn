§6ADRs (architecturally significant decisions)

Only entries that override defaults or have permanent reasoning. Routine commits → ship-log.md.
DateDecisionWhy (load-bearing rationale)
27/04Phase 2 outbox digest for non-meat (aim_outbox + aim_outbox_drain + 1-min cron)Trigger statement aggregation can't span multiple RPC calls. Outbox + cron is the only pattern that groups 12 separate RPC submits into 1 digest msg. Worst-case 90s latency accepted.
25/04Phase 1 statement-level UPDATE trigger for catch_weight (aim_cw_status_summary_v3)Use Postgres REFERENCING OLD/NEW TABLE to aggregate submit_delivery multi-row UPDATE into 1 msg per (item, old→new). Per-row spam was unreadable for 50-bag flows.
25/04Phase A divergence audit (v_sm_cw_divergence + 23:00 cron)sm and CW can drift silently if a trigger misfires. System-driven daily report → Discord alerts before next-day ops.
21/04v_stock_unified as SoT for qty_on_handOne denominator across all UIs. Before this, dashboards disagreed because each computed qty differently.
21/04stock_movements append-only (sm_block_mutation raises on UPD/DEL)Audit trail must be immutable. Errors fixed via compensating insert (see runbook-reconcile-saga.md).
21/04auth.js JWT auto-refresh + 401 retry (ea25058)Session expiry mid-flow caused data loss. One-shot retry with fresh token recovers without UX break.
20/04GitHub Pages + vanilla JS (no build step)Single-store ops + small team. Build chains and SPAs add deploy risk for the ROI.
ongoingSingle Supabase region · no PITR (free tier)Cost. Mitigation = nightly JSON backup + pg_dump pre-op for risky changes.
ongoingTransparent-team RLS (most tables authenticated_all=true)Small team · audit log is the deterrent. Sensitive tables (sm) protected by trigger, not RLS.
Routine commits + ships → see ship-log.md. Don't add <DD/MM> <SHA> <message> entries here.

§7RPC API contracts

Freeze point: changing any signature below = breaking change → bump frontend + bump version.

public.rpc_receive_universal — unified receive

rpc_receive_universal(
  p_actor:     text,     // required
  p_item_id:   uuid,     // required
  p_qty:       numeric,  // > 0 · amount RECEIVED (not running total)
  p_unit_note: text?,
  p_bags:      jsonb?    // only for meat-raw · [{bag_no, weight_g, warehouse}]
): jsonb

Routes by items.type: meat/raw → per-bag catch_weight · else → stock_counts receive event.
Note format written: รับเข้าผ่าน rpc_receive_universal · รับ <qty> <unit> — consumed by aim_stock_counts_trigger (regex รับ\s+([0-9.]+)).

stock.submit_delivery — atomic delivery

stock.submit_delivery(
  p_bill: text, p_branch: text, p_date: date, p_channel: text,
  p_bag_ids: bigint[], p_nm_lines: jsonb
): uuid

Raises: P0001: bag(s) not In Stock · P0001: bill/branch/date required · P0001: no bags and no nm lines

Legacy (do not call from new code)

  • rpc_delivery_out → use submit_delivery

Compatibility rule:
  • ✅ Safe: add optional param at end · add field to return JSON
  • ⚠️ Risky: rename param · change type · make optional → required
  • 🔴 Breaking: remove param · remove return field · change route logic

§9Conventions & Glossary

Phases (from observability buildout, 25-27/04)

  • Phase 1 = noise reduction — statement-level aggregation for catch_weight (Apr 25)
  • Phase 2 = cross-statement aggregation — aim_outbox + cron drain for stock_counts (Apr 27)
  • Phase A = audit-able infra — v_user_actions_daily, v_sm_cw_divergence
  • Phase B = mistake-proof guards — confirm dialogs, yield sanity, audit identity
  • Phase C = self-healing — weekly digest cron Sunday 20:00

Warehouse (คลัง) convention

  • A — incoming raw staging (เพิ่งรับ ยังไม่ผ่าน process)
  • B — cooking / repack staging (ระหว่าง process)
  • C — ready-to-deliver (ปกติใหญ่สุด · ส่งจากตรงนี้ไปหน้าร้าน)

catch_weight status transitions

✅ In Stock ─┬→ 🔄 Repacked   (source consumed in repack)
              ├→ 🚚 Delivered  (shipped via submit_delivery)
              ├→ 🗑️ Disposed   (loss/expired)
              └→ ❌ Out        (cancelled, manual admin)

Blocked by prevent_deliver_if_not_in_stock · Blocked by sm_block_mutation

Actor resolution chain

  1. rpc_receive_universal / submit_delivery → SET app.actor = p_actor
  2. stamp_actor_catch_weight BEFORE INS/UPD → reads GUC → writes cw.actor_id
  3. _resolve_actor() fallback → current_setting('app.actor', true) or auth.jwt()->>'email'
  4. UI: window.nntnCurrentUser (from auth.js) passed as p_actor

Auth refresh cycle (auth.js)

  • On load: check localStorage.nntn_sb_token
  • Expired/near-expiry: auto-refresh via Supabase Auth endpoint
  • On 401: retry once with fresh token (since ea25058 21/04)
  • Not-logged-in: redirect to login.html
